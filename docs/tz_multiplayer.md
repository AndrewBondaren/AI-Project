## Архитектура

`MultiplayerDAGExecutor` реализует тот же интерфейс `DAGExecutorBase` что и `SinglePlayerDAGExecutor`. Переключение — конфиг `engine_mode = "multiplayer"`. Ноды не меняются.

При переходе single → multiplayer: данные совместимы, игрок продолжает без потери прогресса. Требуется рестарт для поднятия персистентного EventBus.

**EventBus в мультиплеере** — персистентный, cross-request, shared across sessions. В отличие от текущего per-request SSE стрима.

## Механика инициативы

**Принцип:** инициатива определяется механикой персонажа, не скоростью набора текста игроком. Текстовая RPG — это про консистентность действий, не про реакцию в реальном времени.

### Фазы хода

**1. Фаза декларации** — игроки объявляют действия в рамках одного тик-окна. Порядок отправки сообщений не влияет на результат.

**2. Фаза разрешения** — движок:
- Парсит action_type, target_uid, duration_ticks из каждого действия (LLM-нода)
- Вычисляет `speed_value` по FormulaNode (ловкость, дистанция до цели, тип действия)
- При конфликте: `speed_A > speed_B` → A выигрывает, B видит результат и получает описание "не успел"

**3. Временная асимметрия** — действие занимает N тиков. Игрок свободен как только его действие завершено — не ждёт самого медленного.

```
Тик 10: A объявляет (длит. 1 тик), B объявляет (длит. 2 тика)
         → конфликт → speed_A > speed_B → A побеждает
Тик 11: A свободен и может ходить. B ещё занят.
Тик 12: оба свободны.
```

**Стратегический выбор:** быстрое действие (1 тик) = чаще ходишь; медленное действие (3 тика) = мощнее, но реже.

## Схема данных (дополнение к основному ТЗ)

```sql
scene_participants (
  session_id,
  character_uid,
  action_available_at_tick,  -- null = свободен прямо сейчас
  PRIMARY KEY (session_id, character_uid)
)

pending_actions (
  id,
  session_id,
  character_uid,
  action_type,
  duration_ticks,
  target_uid,           -- nullable
  declared_at_tick,
  speed_value,          -- вычислено FormulaNode при разрешении
  status                -- 'pending' | 'resolved' | 'cancelled'
)
```

В single-player: `pending_actions` не нужна, `action_available_at_tick` всегда null.

## Риски реализации

**1. Парсинг естественного языка → дискретное действие**
LLM извлекает только `action_type + target_uid`. Механические параметры (`duration_ticks`, `speed_formula`) — движок читает из `action_registry` по `action_type`. LLM не знает о механике.

```
"Я атакую стражника"
    ↓ LLM (action detection нода)
{ action_type: "attack", target_uid: "guard_001" }
    ↓ Engine lookup
action_registry["attack"] → { duration_ticks: 1, speed_formula: "agi_speed" }
    ↓ Engine compute
speed_value = formula(character.stats)
```

Реальный риск сужается до: действие игрока не мапится ни на один `action_type` в реестре → repair loop или дефолтное нарративное действие.

**2. Фаза декларации — таймаут**
Сколько реального времени ждём пока все игроки объявят действие? Варианты:
- Фиксированный таймаут → автоматическое разрешение
- "Объявил — обрабатывается независимо; конфликт разрешается если пересеклись"
Второй вариант проще для асинхронной текстовой RPG. Нужно решить.

**3. Определение конфликта**
"Я атакую стражника" + "Я подкупаю стражника" — семантически конфликт, механически разные action_type и разные цели в системе. LLM должен распознавать пересечения. Нетривиально.

## Мультипротагонист (два персонажа в одном мире)

Потенциальная кор-механика. Архитектура `UNIQUE(world_id, player_character_id)` закладывает фундамент.

Открытые вопросы:
- Кто "ведущий" персонаж в сцене? LLM нужна точка зрения.
- NPC который для A — враг, для B — союзник.
- Чьи намерения двигают историю в один ход?
- Как два игрока взаимодействуют друг с другом (не через NPC)?

Полный дизайн — отдельная задача после базового мультиплеера.
