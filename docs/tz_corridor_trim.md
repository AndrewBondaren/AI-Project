# ТЗ: Обрезка коридоров до последней значимой точки

## Проблема

Коридорные комнаты (gallery_n, gallery_e и подобные) генерируются со случайной длиной (`depth_range` / `width_range`).
К ним прикрепляются комнаты через `attach_to`. Если прикреплённых комнат оказалось меньше, чем позволяет длина коридора,
его дальний конец образует мёртвый тупик — закрытый участок без выходов, который пользователь не контролирует.

**Пример:** `gallery_e` depth=14, прикрепилась 1 комната (y=10..16), коридор уходит до y=23 — тупик y=16..23.

Это ответственность генератора: случайность длины не должна создавать неиграбельные участки.

---

## Ключевое допущение

**Коридоры всегда начинаются от лестницы.** Это инвариант логики построения уровней.

Следствие:
- **Start-конец** коридора = конец, ближайший к якорю своей лестницы (всегда известен)
- **Face-направление** = противоположная сторона (туда растёт коридор)
- Определять facing из геометрии не нужно — он задан связью staircase → corridor в шаблоне

---

## Алгоритм

**Фаза:** после `layout_level` (mode_b завершён), до генерации ячеек.

**Применяется к:** комнатам с `room_type = "corridor"` (или любой тип, у которого есть `attach_to`-дети).

### Шаги

1. **Определить start-конец**
   - Найти якорь лестницы, которая связана с этим коридором (через `staircases[*].stops`)
   - Start-конец = конец коридора вдоль главной оси, ближайший к этому якорю

2. **Определить face-конец**
   - Противоположный конец от start вдоль главной оси

3. **Идти от face к start**, на каждой позиции P проверять:
   - **Прикреплённая комната:** есть ли размещённая комната с `attach_to == corridor.room_id`,
     чья стена пересекает коридор на слайсе P?
   - **Якорь другой лестницы:** есть ли `fr_anchor` / `to_anchor` любой другой лестницы
     в радиусе 2 клеток от стенок коридора на позиции P?
   - При первом найденном → `last_valid_pos = P`, остановиться

4. **Обрезать:**
   - Если `last_valid_pos` найден:
     `new_length = |last_valid_pos - start_coord| + 1 + 1` (последний +1 — стена закрывающая)
   - Если не найден:
     `new_length = MIN_CORRIDOR_LENGTH` (константа, например 3)
   - Если `new_length < current_length` → применить trim

5. **Trim** = уменьшить `room.depth` или `room.width` (в зависимости от главной оси).
   Origin комнаты не меняется (start-конец фиксирован).

---

## Граничные случаи

| Ситуация | Поведение |
|---|---|
| Коридор соединяет две лестницы | face-конец содержит якорь другой лестницы → trim не применяется |
| Ни одна комната не прикрепилась | trim до MIN_CORRIDOR_LENGTH от start |
| Все прикреплённые комнаты у start-конца | face-сторона тупиковая → trim до последней комнаты |
| new_length >= current depth | Trim не нужен, ничего не делаем |
| Коридор квадратный (width == depth) | Не является коридором — не обрабатываем |

## Место реализации

После вызова `layout_level(...)` в `structureGeneratorService._layout_rooms()`,
перед переходом к следующему z_offset.

```python
# После layout_level(level_rooms, ...)
_trim_corridor_rooms(level_rooms, all_placed_by_id, template, staircase_anchors)
```

Функция `_trim_corridor_rooms` принимает:
- `level_rooms` — комнаты текущего уровня (уже размещённые)
- `all_placed_by_id` — словарь всех размещённых комнат (для поиска attach-детей)
- `template` — для поиска staircase stops
- `staircase_anchors` — список якорей лестниц (fr_anchor / to_anchor), которые уже известны к этому моменту

## Открытые вопросы

1. **MIN_CORRIDOR_LENGTH** — сейчас предлагается 3. Нужно ли делать настраиваемым в шаблоне?
2. **Обрезка с обоих концов?** — Текущий алгоритм обрезает только face-конец.
   Теоретически start-конец тоже может иметь пустой буфер, но практически это маловероятно
   (лестница сразу у start). Пока не делаем.
3. **room_type фильтр** — применять только к `"corridor"`, или к любой комнате с `attach_to`-детьми?
   Пока только `"corridor"` — безопаснее.
