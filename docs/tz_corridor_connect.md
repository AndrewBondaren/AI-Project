# ТЗ: Соединение изолированных коридоров пост-обработкой

## Проблема

После обрезки (см. `tz_corridor_trim.md`) два коридора на одном уровне могут оказаться
пространственно несвязанными. Например, `gallery_n` и `gallery_e` — оба начинаются от своих
лестниц, но между собой не соединены ни одним проходом.

Пользователь не контролирует это напрямую: связанность коридоров зависит от случайных размеров
и размещения комнат. Генератор обязан обеспечить связность там, где это возможно.

---

## Порядок выполнения в пайплайне

```
layout_level (mode_a + mode_b)
    ↓
trim_corridors          ← до ячеек (меняет _RoomInstance размеры)
    ↓
build_level_cells
    ↓
connect_corridors       ← после ячеек (добавляет ячейки напрямую + wall breach, как UndergroundTunnelBuilder)
    ↓
passages
    ↓
wall_openings
```

## Архитектура: общий pathfinder

В `undergroundTunnel.py` уже реализована функция `_bfs_min_turns` — 0-1 BFS с минимизацией поворотов.
Она должна быть вынесена в отдельный класс `TunnelPathFinder` и переиспользоваться обоими потребителями:

| Потребитель | Модуль |
|---|---|
| Подземный тоннель | `staircase/undergroundTunnel.py` → рефакторинг |
| Соединение коридоров | новый `corridorConnector.py` |

**Интерфейс `TunnelPathFinder`:**
```python
class TunnelPathFinder:
    def find_path(
        self,
        start:   tuple[int, int],
        target:  set[tuple[int, int]],
        blocked: set[tuple[int, int]],
        bounds:  tuple[int, int, int, int] | None = None,
    ) -> list[tuple[int, int]]:
        ...
```

**Pathfinder работает только в XY.** Z-ограничение не является параметром pathfinder'а —
оно учитывается вызывающим кодом: caller вычисляет `blocked` с учётом Z-диапазона тоннеля
(исключает ячейки, над которыми на z-уровнях выше уже что-то есть) и передаёт уже готовый blocked set.
Логика одинакова для подземного тоннеля и соединения коридоров.

`UndergroundTunnelBuilder` рефакторится на использование `TunnelPathFinder`.
Логика размещения ячеек остаётся в самих builder'ах — pathfinder только находит маршрут.

**После рефакторинга:** в `UndergroundTunnelBuilder` добавить арки в точках соединения
(сейчас там plain wall breach без арок).

## Алгоритм

### Шаг 1: Проверка связности

Строим граф коридоров уровня:
- **Узлы:** все размещённые комнаты с `room_type = "corridor"` на уровне
- **Рёбра:** два коридора связаны, если между ними есть путь через граф соединений
  (archway / doorway — включая транзитные через промежуточные комнаты)

Если все коридоры уровня в одной связной компоненте → алгоритм не запускается.

### Шаг 2: Поиск пути между несвязанными коридорами

Для каждой пары несвязанных коридоров (A, B):

**Область поиска** — XY-клетки в пределах здания:
- В границах здания = внутри bounding box всех размещённых комнат уровня (или их объединённого footprint)
- **Запрещено:** клетки, входящие в footprint любой комнаты
- **Запрещено:** клетки за пределами здания
- **Разрешено:** свободное пространство между комнатами

**Старт и цель:**
- Старт = клетки внешней стены коридора A (все, не только face-конец)
- Цель = клетки внешней стены коридора B

Алгоритм поиска: BFS или A* по свободным клеткам.

### Шаг 3: Создание соединительного прохода

Если путь найден:
- Создаём новый `_RoomInstance` типа `corridor` вдоль найденного пути
- Добавляем соединительный проход в коллекцию комнат уровня
- Он участвует в дальнейшей генерации ячеек наравне с остальными комнатами

Если путь не найден → ничего не делаем, логируем warning.

---

## Правила построения соединительного прохода

- Ширина: минимально возможная — **1 клетка интерьера** (3 с учётом стен), если пространство позволяет
- Не может накладываться на существующие комнаты
- Соединяется с обоими коридорами через archway (пробивка стены)

---

## Параметры соединительного тоннеля

**Высота (z_height)** и **ширина** берутся из коридора A (родителя пути).

Каскадный fallback по ширине — срабатывает если путь не найден:

1. Ширина = ширина коридора A
2. Ширина = 1 клетка интерьора
3. Путь не строим, логируем warning

---

## 3D-проверка свободного пространства

Тоннель занимает трёхмерный объём: `[x..x+w-1] × [y..y+d-1] × [z_floor..z_floor+z_height-1]`.

При проверке каждой клетки маршрута **(x, y)** проверяем весь z-диапазон тоннеля:
- Нельзя перезаписывать ячейки на z-уровнях выше — если в диапазоне `[z_floor, z_floor+z_height-1]`
  уже есть клетки других комнат (шахты, комнаты с нависанием, комнаты уровня выше) → клетка **(x, y)** недоступна для маршрута

Это применяется ко всем трём измерениям тоннеля:
- **Высота:** если на z-уровнях выше что-то есть — текущий z_height не помещается в эту ячейку
- **Ширина:** если при полной ширине хотя бы одна из W клеток имеет конфликт по z → пробуем fallback ширину
- **Длина:** при прокладке пути BFS/A* просто не включаем в граф конфликтные ячейки

Fallback по ширине применяется ко всему маршруту целиком (не по ячейкам): сначала пробуем весь путь с шириной A, потом весь путь с шириной 1.

## Старт пути

Все внешние клетки коридора A (клетки, примыкающие к внешним стенам снаружи и находящиеся
в свободном пространстве) являются кандидатами на точку выхода.

Кандидаты **сортируются по расстоянию от face-конца** вдоль главной оси:
торцевая стена (face wall) — первой, затем боковые стены от face к start.

```
face-конец  [ торец | бок-лев | бок-прав | бок-лев | бок-прав | ... ] start-конец
                ↑ пробуем в этом порядке, первый успех → строим
```

Для каждого кандидата запускаем BFS к target-клеткам коридора B.
Первый кандидат, у которого BFS нашёл путь → используем.
Если ни один не дал пути → тоннель не строим, логируем warning.

---

## Определение "соединены"

Два коридора считаются **связанными** только при наличии прямого прохода коридор→коридор
(archway или doorway между ними). Транзит через комнату категорически не считается связью.

---

## Границы здания для pathfinding

Используем уже вычисленный `level_footprint_bounds[z_offset]` = `(min_x, min_y, max_x, max_y)` —
bounding box всех размещённых комнат уровня. Вычисляется в `_layout_rooms()` и передаётся
в `_connect_corridors()` как готовый параметр.

**Свободное пространство** = клетки внутри этого bounding box, не входящие ни в один `room.get_footprint()`.

---

## Приоритет и порядок соединений

Если несвязанных коридоров несколько — соединяем **последовательно, по возрастанию расстояния**:

1. Вычисляем расстояние между всеми несвязанными парами — **минимальное расстояние между ближайшими стенами** двух коридоров
2. Берём ближайшую пару (A, B) → соединяем
3. После соединения пересчитываем граф связности — новый тоннель может транзитно связать другие пары
4. Повторяем до тех пор, пока все коридоры не связаны или пути больше не находятся

## Точка соединения: wall breach

Место, где новый тоннель примыкает к целевому коридору B, оформляется как **wall breach** —
пробоина в стене (существующий механизм `wallBreach`), а не door/archway.

Это касается обоих концов тоннеля:
- Выход из коридора A → wall breach в стене A
- Вход в коридор B → wall breach в стене B

Wall breach без колонн и арок — просто пробоина в стене, без декоративных элементов.
